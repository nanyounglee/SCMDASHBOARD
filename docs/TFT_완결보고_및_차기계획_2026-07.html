이번 범위에서 제외·미달한 것

범위 한계전사 프로세스 미포함

  • 앞단 — 매출·수요계획(전사 S&OP, 판매·운영 계획) 연계가 없습니다. 매출은 수동 업로드 근사치만 반영됩니다
  • 뒷단 — 납기(입하) 이후의 검수 방식 → 임가공(입고 후 추가 가공) → 고객납품에 이르는 일정·비용 데이터가 빠져 있습니다. 이로 인해 전반적인 제조원가에 대한 현재 데이터의 신뢰 오차 범위가 큽니다

데이터 부재구현하지 못한 기능

  • 이슈 예방·예측 지표 없음 — 현재 지표는 100% 사후 집계입니다. 문제를 사전에 알려주는 리드 인디케이터(선행 경고 지표)의 추이 데이터가 원천에 없습니다
  • 재고 분류체계 부재 — ⓐ 직접제조 선매입재고 ⓑ 단순조달 협력사 소유 재고 ⓒ 재고는 아니지만 협력사 상시 재고를 활용해야 하는 제품, 3구분의 분류 기준이 없어 "협력사 상시 재고량 파악 대상"을 시스템이 식별할 수 없습니다
  • 제품별 안전 상시 재고량 기준 부재 — 산정 체계와 협력사 재고량 수집 채널이 없습니다
  • IQC(입고 품질검사) 전수 데이터 없음(입하 표본 94건 중 검수기록 0건) · CAPA(시정·예방조치 — 이슈 원인을 기록하고 재발을 막는 관리 루프) 미구축 — 수집 방안 설계는 완료되어 실행만 이월된 상태입니다

기술 구조정적 HTML의 한계

  • 단일 651KB HTML — 양방향 입력이 안 되어, 쓰기는 개인 브라우저 임시 저장 또는 GitHub 커밋(토큰 보유자만)의 우회 구조입니다
  • GitHub Pages 공개 URL — 접근 통제가 없습니다. 저장소가 공개 상태인 한 발주액·원가 CSV 원본까지 외부에 노출됩니다 (→ §03 보안 조치 필수)
  • 차트·PDF 라이브러리가 외부 CDN 의존이라 오프라인 제약이 있고, 데이터 증가 시 브라우저 성능 한계가 있습니다

03실행 지원 요청 — 기술·자원

다른 회사라면 전담 IT·DX 부서가 담당할 영역을 현재 TFT 팀원들이 현업과 겸직으로 수행하고 있습니다. 차기 프로젝트가 성립하려면 기술 옵션과 함께 아래의 실질적인 자원 지원이 필요합니다.

A. 배포·보안 옵션

현 상태 진단. 대시보드는 GitHub Pages 공개 URL로 서빙되며 접근 통제가 없습니다. 저장소가 공개인 한 발주액·협력사·원가 CSV 원본까지 외부 노출 상태입니다. 어떤 옵션이든 저장소 비공개 전환 검토가 1순위이며, 무료 요금제에서는 비공개 저장소의 Pages도 결국 공개 URL이므로 접근 통제를 병행해야 합니다.
옵션방법비용소요평가
A. 도메인 + Cloudflare Access 권장·단기 도메인 구매 → Cloudflare Pages로 정적 파일 이전 → 접근통제 서비스(Zero Trust Access)로 회사 이메일 도메인/명단만 허용(이메일 인증 또는 Google 로그인) 도메인비 연 1~2만원
(접근통제 무료 50석)		1~2일 현행 정적 구조를 유지한 채 "인트라넷처럼" 쓰는 요구에 정확히 부합합니다 — 도메인 구매 시나리오
B. GitHub Enterprise Pages Enterprise 요금제 조직이면 Pages를 조직 구성원 한정 공개로 설정 인당 월 $21~설정만 회사가 이미 Enterprise면 최선, 아니면 비용 과다
C. Netlify/Vercel 비밀번호 정적 배포 + 사이트 공용 비밀번호 유료 요금제반나절 간단하지만 공용 비밀번호라 개인 식별·권한 회수가 안 됩니다
D. 사내 서버/NAS 사내망 웹서버에 정적 파일 배치 IT 자원IT 협조 진짜 내부망입니다. 다만 재택·외부 접근이 안 되고 배포 자동화를 따로 만들어야 합니다
E. 로그인 웹앱 전환 차기 WS1 Supabase/Firebase 등 인증+DB 서비스 기반 재구축 무료~소액차기 본 과제 양방향 입력·권한·이력의 근본 해결 — 단기 조치가 아닌 차기 본류입니다

권장 로드맵 — 즉시(2026.8): 저장소 비공개 전환 검토 + 옵션 A 적용. 주간 자동 커밋 파이프라인은 배포 대상만 바뀌므로 유지됩니다. 차기 착수(2027.3): WS1에서 옵션 E로 전환하며, 옵션 A에서 마련한 도메인·접근통제는 신규 웹앱에 그대로 재사용합니다.

B. 개발 인력 지원 — 외주 포함

이번 이니셔티브는 AI 도구를 활용해 TFT가 자체 구축했지만, 차기 WS1(로그인·DB 웹앱 전환)은 보안·인증이 걸린 전문 개발 영역입니다. 다음 중 하나 이상의 인력 지원이 필요합니다.

방안내용예상 규모
외주 개발 권장개발사 또는 프리랜서 개발자에게 WS1 웹앱 구축을 위탁하고, TFT는 요구사항 정의·검수를 담당개발자 1~2인 × 3~4개월 수준 (표준 웹앱+로그인+DB 범위)
내부 IT 인력 협업사내 IT·개발 조직이 있다면 기간 한정 파견 또는 겸직 배정1인 × 프로젝트 기간
자체 구축 + 외부 검토 최소안AI 도구 기반으로 TFT가 구축하되, 보안·인증 설계와 최종 코드는 반드시 외부 전문가 검토를 거침검토 용역 2~3회 (착수 설계·중간·오픈 전)

어느 방안이든 보안·인증 부분만큼은 외부 전문가의 검토가 필수라는 점을 강조드립니다. 사내 데이터가 올라가는 시스템을 비전문가 검증만으로 여는 것은 §03-A에서 지적한 노출 문제를 형태만 바꿔 반복하는 일이 됩니다.

C. 도구·운영 비용 — 회사 지원 필수

이번 TFT는 AI 도구(Claude 유료 계정) 비용을 담당자들이 사비로 부담하며 업무를 진행했습니다. 업무 수행에 필수적인 도구 비용을 개인이 부담하는 구조는 지속 가능하지 않고 형평에도 맞지 않습니다. 회사 규정과 별도로, TF 운영 시에는 아래 도구·운영 비용의 회사 지원이 필수적입니다.

04TFT 팀원 역량 개발 제안

차기 착수 전 7개월(2026.8 ~ 2027.2)을 학습 기간으로 활용하면, 외주를 쓰더라도 요구사항 정의와 검수를 내부에서 주도할 수 있습니다. 업무 시간 내 주 1~2시간의 팀 스터디를 회사가 공식 인정해 주는 것을 함께 요청합니다.

05TFT 논의·결정 요청 사항

  1. 차기 프로젝트 범위 승인 — WS1~WS5 구성과 6개월 일정(2027.3~8)의 타당성을 검토해 주시기 바랍니다
  2. 데이터 확보 협의 주체 지정 — 재고 소유구분 분류(구매전략·외주생산 공동), 검수·임가공·납품 데이터(물류 부서), 수요계획(전사 S&OP)
  3. 보안 단기 조치 승인 — 도메인 구매 + 접근통제 적용, 저장소 비공개 전환 (즉시 실행 권장)
  4. 개발 인력 지원 결정 — 외주 개발 위탁 / 내부 IT 협업 / 자체 구축+외부 검토 중 택일과 예산 검토
  5. 도구·운영 비용의 회사 부담 확정 — AI 도구 구독 등 필수 도구의 사비 부담 구조 해소
  6. 착수 전 준비 담당 배정 — IQC Forms 가동, CAPA 원인분류 확정, 재고 3구분 초기 분류, 타 부서 인터뷰 1라운드
