# Airtable 품질관리 필드 추가 명세서

> 작성: 2026-07-20 · SCM 대시보드 품질관리 기능 확장(CAPA·재발분류·FAI·IQC)을 위한 Airtable 스키마 변경 요청서
> 대상: Airtable 관리자(외주생산파트) · 변경 후 대시보드 반영은 자동(기존 주간/수동 CSV 파이프라인)

## 왜 필요한가 (요약)

대시보드 품질 지표는 현재 100% 사후 집계입니다. 아래 필드를 추가하면 기존 CSV 자동 커밋 파이프라인에 **그대로 실려** (스크립트 수정 없이 CSV 컬럼으로 유입) 다음이 가능해집니다:

| 추가 대상 | 가능해지는 기능 | 입력 부담 |
|---|---|---|
| ① 이슈 테이블 `수주처` lookup | 모든 협력사 품질 지표의 조인 정확도 개선 | **0 (자동)** |
| ② 이슈 테이블 원인분류 1개 | 재발 이슈 판정(동일 원인 반복) | 이슈당 클릭 1회 |
| ③ 이슈 테이블 CAPA 필드 4개 | CAPA 트래커(기한 초과 자동 하이라이트) | 이슈당 2~3회 입력 |
| ④ FAI 신규 테이블 | 초도품 승인 대장 | 초도 건당 1행 (연 10~15건 예상) |

---

## ① 이슈 테이블 — `수주처` lookup 필드 (최우선, 입력 부담 0)

이슈 레코드는 movement 기반이라 발주(task) 레코드와 링크되어 있을 가능성이 높습니다.

| 항목 | 값 |
|---|---|
| 필드명 | `수주처 (from order)` |
| 타입 | **Lookup** — 연결된 order/task 레코드의 `수주처` |
| 입력 | 자동 (lookup이므로 수기 입력 없음) |

- 현재 대시보드는 이슈→협력사를 프로젝트 코드 접두어로 **추정 조인**하며, 한 프로젝트에 협력사가 여럿이면 첫 협력사에만 귀속되는 실측 왜곡이 있습니다 (ARCHITECTURE §4-12).
- 이 필드가 CSV에 실리면 대시보드가 추정 조인 대신 정식 값을 우선 사용하도록 바꿀 수 있습니다 (협력사 이슈율·상세 모달·재발 뷰·유출률 전부 정확도 상승).
- **만약 이슈 레코드가 order와 링크되어 있지 않다면** 이 방식이 불가하므로 알려주세요 — 대안(프로젝트 코드 정규화)을 설계합니다.

## ② 이슈 테이블 — 근본원인 분류 (재발 판정용)

| 항목 | 값 |
|---|---|
| 필드명 | `근본원인분류` |
| 타입 | Single select |
| 옵션(초안) | `원자재` / `인쇄·컬러` / `공정·가공` / `포장·구성` / `설계·사양` / `물류·취급` / `검수누락` / `기타` |
| 입력 시점 | SCM 담당자가 이슈 대응 판단 시 (기존 `품질등급의견_SCM` 기입과 같은 타이밍) |

- 기존 `품질이슈내용구분`은 내용 구분이지 원인 분류가 아니라 재발 판정에 쓸 수 없습니다.
- 옵션 명칭은 팀 용어에 맞게 바꿔도 됩니다 — 대시보드는 값을 그대로 그룹핑하므로 어떤 명칭이든 동작. 단 **한번 정하면 바꾸지 않는 것**이 이력 일관성에 중요합니다.

## ③ 이슈 테이블 — CAPA(시정조치) 필드 4종

| 필드명 | 타입 | 비고 |
|---|---|---|
| `CAPA상태` | Single select: `접수` / `원인분석` / `대책수립` / `효과검증` / `종결` / `해당없음` | 기본값 `접수` 권장 (Airtable 자동화로 신규 레코드에 자동 세팅 가능) |
| `대책기한` | Date | 협력사 회신 요구 기한 — 기한 초과 시 대시보드가 자동 하이라이트 |
| `종결일` | Date | `CAPA상태`=`종결` 전환 시 기입 (Airtable Automation으로 자동 스탬프 가능) |
| `협력사대책` | Long text | 협력사가 회신한 대책 원문 — 기존 `*_대응방안_SCM` 3종(내부 판단)과 별개 |

- 경미한 운영/수량 이슈까지 CAPA를 강제하면 입력 부담이 커집니다 — `해당없음` 옵션으로 선별 운영하거나, `품질이슈대응필요여부`(기존 필드)=필요 건만 CAPA 대상으로 삼는 운영 규칙을 권장합니다.
- **갱신 주기**: 기한 관리 특성상 주 1회로는 부족 → 이슈 CSV만 매일 갱신하는 별도 워크플로를 대시보드 저장소에 추가 예정 (`AIRTABLE_SOURCES`에서 issue 항목만 담은 일일 잡 — 코드 수정 없이 가능함을 확인 완료).

## ④ FAI(초도품 승인) 신규 테이블

새 테이블 `초도품승인` (같은 SCM KPI 베이스 내):

| 필드명 | 타입 | 비고 |
|---|---|---|
| `승인번호` | Autonumber 또는 Formula | PK |
| `제품` | Text (또는 goods/parts 링크) | 대시보드 자동 추출값 복사 |
| `협력사` | Text (또는 협력사 테이블 링크) | |
| `초도구분` | Single select: `신제품` / `신규협력사` / `공정변경` / `재개` | |
| `첫발주일` | Date | 대시보드가 발주 RAW에서 자동 검출한 날짜 |
| `승인상태` | Single select: `대기` / `조건부승인` / `승인` / `반려` | |
| `판정일` | Date | |
| `판정자` | Single select 또는 Collaborator | |
| `한도견본` | **Attachment** | 승인 샘플·한도견본 사진 |
| `비고` | Long text | 조건부 승인 조건 등 |

- 후보 자동 검출은 대시보드가 담당: 2025+2026 발주 데이터 실측 결과 2026년 신규 제품×협력사 조합 **177개**, 신규 협력사 **16개** 검출 확인. 교정시트·샘플 발주 이력(연 10~15건)과 교차하면 실제 승인 프로세스 흔적과 연결됩니다.
- CSV 파이프라인 등록: `AIRTABLE_SOURCES`에 `{"key":"fai","base":"...","table":"초도품승인","view":"...","file":"fai.csv"}` 항목 1개 추가 + 대시보드 로드 키 등록(코드 소량).

## ⑤ 확인 요청 — movement 테이블 검수 데이터 (IQC 전수화)

현재 이슈 CSV의 `검수수량`/`불량수량_샘플링검수`는 **이슈가 발생한 행에만** 있어(344행 중 각각 164행·40행) 검수합격률 KPI가 "이슈 건 중 합격률"로 왜곡되어 있습니다. 이슈 레코드가 movement 단위인 것을 보면 **movement(산출이동) 테이블에 전수 검수 기록이 있을 가능성**이 있습니다.

**확인 방법**: GitHub 저장소 → Actions 탭 → `airtable-discover` → Run workflow (커밋 없는 읽기 전용 — 토큰 접근 가능한 모든 Base/Table/View 이름이 로그에 출력됩니다). 로그에서 movement/산출이동 계열 테이블을 찾아:
- **전수 검수 기록이 있으면**: 해당 테이블에 뷰 하나 추가 + `AIRTABLE_SOURCES` 항목 1개 → 새 입력 프로세스 없이 IQC 전수 지표 확보 (분류 Ⓑ)
- **없으면**: 협력사 출하 전 자가검사 성적서 제출용 Airtable Form 설계로 전환 (분류 Ⓒ — 별도 논의)

## 반영 후 대시보드 작업 (참고 — 저장소 쪽에서 진행)

1. 이슈 CSV 헤더에 새 컬럼 반영 (`fetch_airtable_sources.mjs`는 기존 CSV 헤더를 재사용하므로, 필드 추가 후 첫 1회는 `CSV/issue.csv` 헤더에 컬럼을 추가해줘야 새 필드가 실리기 시작 — 저장소 쪽에서 처리)
2. CAPA 트래커 탭 · 재발 뷰 원인분류 축 추가 · FAI 대장 화면 구현
3. 일일 이슈 갱신 워크플로 추가
