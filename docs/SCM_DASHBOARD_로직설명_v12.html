3. 핵심 파싱 로직

3-1. 발주 TASK 카운트 (KPI #1)

기준: 과업지시일자 월 기준 행 카운트 재고생산 판별: movement_산출이동에 '재고' 포함 또는 project에 '재고' 포함 프로젝트 발주 = 전체 - 재고생산 긴급 발주: 긴급여부 === '긴급' AND 수주처 !== '서울디지털인쇄협동조합'

3-2. 월 매입금액 (KPI #2) — 세금계산서 분할 처리

function purchaseByMonth(row): 1. 세금계산서작성월 (from 지출결의) 파싱 → 월 배열 2. 단일 월: 공급가액 전체 배정 3. 분할 월: 합계_분할 필드의 월별 금액 사용, 없으면 균등 분할

3-3. 입하경과일 자동 산출 (하도급법 연동)

function calcElapsedDays(row): 기준일 = 입하확정일 (from movement_산출물) || 입하예정일 (from movement_산출물) 종료일 = 실제입하일 (from movement) || 오늘 경과일 = (종료일 - 기준일) / 86400000 하도급법 위험 판별: 공급망_RAW의 하도급계약 대상여부 === '대상' AND 미입하 (실제입하일 비어있음) AND 경과일 >= 90일

3-4. 원가 자동 산출 (파츠마스터 + 공정DB · 원가분석 화면용)

1. sync_itemdb 필드에서 파츠코드 추출 (정규식: [A-Z]{4}_PT\d+) 2. 파츠코드 → parts_master.json에서 수량별 단가 조회 3. 공정DB → 굿즈코드별 공정 구성 + 표준단가 조회 4. 제품원가 = Σ(파츠별 구간단가 × 수량) 5. 마진율 = (판매가 - 원가) / 판매가

3-5. 실패비용 산출

function getReworkCostForProject(projName): 발주_RAW에서 project가 동일한 프로젝트의 행 중 task_id에 '재제작' 문자열이 포함된 행의 취득원가 합계 적용 위치: 이슈 현황, 고객인지이슈 모달, 이슈 모달, KPI 실패비용율(§3-15)

3-6. 전년 동기 비교 (YoY) / 3-7. 전월 대비 (MoM)

YoY: data_2025.json에서 전년 동월 사전집계 데이터 조회 (CSV 업로드 불필요) MoM: getPrevMonth('2026.6') → '2026.5', getPrevMonth('2026.1') → '2025.12'

3-8. 품절 성격 분류

비의도적 = 품절성격.includes('비의도적') 의도적 = 품절성격.includes('의도적') AND NOT includes('비의도적')

3-9. 재고운영 Agg 로직 상세 v10

3-10. 협력사 평가 점수 산식 (원본, 5축 15점)

협력사 QCD/관계 원점수는 아래 5축 산식으로 산출하며, v9(반기환산)·v12(Tier 분류 교체) 모두 이 산식 자체는 그대로 재사용한다.

평가 항목 (총 15점 만점, 반기 1~6월 데이터 기준): TASK 점수 (3점) — 반기 ≥200건→3, ≥50→2, ≥6→1 (연 400/100/11 환산) 매입대금 (3점) — 반기 ≥5,000만→3, ≥2,500만→2, ≥1원→1 (연 1억/5천만 환산) 이슈대응 (3점) — 이슈율 점수×70% + 미입하율 점수×30% 예외사항 (3점) — 정성평가 (발주담당 분기 기입) 대체가능 (3점) — 정성평가 (발주담당 분기 기입) A/B/C/D 등급(변경 없음): A >=12 / B 6~11 / C 3~5 / D 0~2 1Q/2Q 등급이 다르면 두 분기 총점 평균으로 재산정

3-11. 발주 상세 분석 3-tab + DB원가 비교 v11 신규

openModal('orderTask')를 renderOrderTaskModal()로 리팩터링, 탭 3개로 구성된다.

① 발주현황 Top — renderOrderTaskTop(): 기존 굿즈 발주량 내림차순 로직 그대로 유지 ② 미발주현황 — computeNoOrderStatus(): 굿즈코드 단위로 D.order 전체 히스토리에서 "마지막 발주월" 확인 → 선택월(SEL_MONTH)에 발주가 없으면 미발주로 집계 (과거 발주 이력이 있는 품목만 대상 — 비스포크/신제품은 제외) ③ DB원가비교 — 3단계 계산: 1) getGoodsPartsIndex(): '굿즈 연결 현황 (from item)' 컬럼(콤마구분)으로 굿즈명 → 구성 파츠 배열의 역인덱스(BOM) 구성 2) partTierCost(row, qty): 파츠 1개의 수량구간별 표준원가 중 "발주수량 이하의 가장 큰 구간" 단가를 채택. 해당 구간이 비어있으면(blank) 다음으로 큰 구간으로 폴백. 단, 구간 값이 0으로 명시된 경우는 유효한 0원으로 인정(빈칸과 구분). 3) computeGoodsDbCostForOrder(goodsName, qty): 굿즈를 구성하는 모든 파츠(Product Parts + Package + Printing 전 유형)의 구간단가를 합산한 "구성파츠 단가합" × 발주수량 = 해당 오더의 DB원가 → 월별로 합산해 실제 매입금액(공급가액)과 비교 → 절감율 = (DB원가 - 매입금액) / DB원가 × 100
검증: 실제 parts.csv 대상 샘플 수량(1/100/220/5000개)으로 Python 시뮬레이션 대조 검산 완료. 예: 200개 발주 시 100개 구간 단가 적용(발주수량 이하 최대 구간 규칙), 파츠 유형 3종(Product/Package/Printing) 전부 합산.

3-12. 공급망 포트폴리오 — QCD×관계 Tier v12, 분류 기준 교체

§3-10의 5축 점수 산출 로직은 그대로 두고, 4분면 분류 기준만 교체했다. 기존(v9) 매입액×등급 4분면(Core/Strategic/Preferred/General)을 폐기하고, QCD×관계 점수 기준 Tier로 바꿨다.

QCD (9점) = TASK 점수 + 매입대금 점수 + 이슈대응 점수 관계 (6점) = 예외사항 점수 + 대체가능 점수 Tier1 · Core : QCD>=7 AND 관계>=5 Tier2 · Performer : QCD>=7 AND 관계<5 Tier3 · Developing : QCD<7 AND 관계>=5 Tier4 · General : QCD<7 AND 관계<5 ※ A/B/C/D 등급(§3-10, 총점 기준)은 폐지되지 않고 Tier와 별도로 함께 표시된다. Tier는 "협력 구조" 관점, 등급은 "종합 성과" 관점으로 서로 보완한다.
