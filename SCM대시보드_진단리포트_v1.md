# SCM 대시보드 진단 리포트 v1

**무슨 일이 있었는지:** `SCMDASHBOARD/index.html`(v6, 4,145줄) 전체를 정독하고 실제 CSV 8종으로 계산 로직을 재현·대조한 결과, **상위보고에 잘못된 숫자로 나갈 치명 버그 4건**과 **공개 즉시 빈 화면으로 터질 배포 구조 문제 2건**을 확인했습니다. 특히 "의도적 품절" 건수는 지금 **52건으로 표시되지만 실제는 35건**이고(+17건 과다), 굿즈코드는 전 화면에서 사라져 있으며(잘못된 컬럼 참조), 재고운영·EOQ·졸업 3개 페이지는 현재 CSV로는 통째로 비어 나옵니다. 아래는 근거와 함께 정리한 전체 목록이며, 마지막에 v7 핫픽스 제안을 뒀습니다. 코드 수정은 승인 후 진행합니다.

---

## A. 치명적 — 상위보고에 틀린 숫자가 나감 (즉시 수정)

### C1. "의도적 품절" 건수 과다 집계 (+17건)
- **위치:** `updateStockSection` L1618, `renderStockoutDetail` L2568·L2582
- **원인:** `intent = sl.filter(r => (r['품절 성격']||'').includes('의도적'))` — 문자열 `'비의도적'` 안에 `'의도적'`이 포함되므로, 비의도적 17건이 의도적 카운트에 전부 섞임.
- **데이터 검증:** 품절리스트 54행 → 비의도적품절 17 / 의도적품절 35 / 공란 2. 코드 산출 intent=**52**(=35+17), 실제 순수 의도적=**35**.
- **영향:** 품절 상세·재고 현황 KPI 카드에 "의도적 품절 52"로 표시. 대응 우선순위 판단(비의도적=즉시대응)까지 흐려짐.
- **수정:** `n.includes('의도적') && !n.includes('비의도적')`.

### C2. 굿즈코드가 전 화면에서 누락 (잘못된 컬럼 참조)
- **위치:** `goodsCode()` L969 — `r['sync_item이니셔티브']`
- **원인:** 발주 RAW에 그 컬럼이 **없음**. 실제 컬럼명은 `sync_itemdb`.
- **데이터 검증:** `sync_item이니셔티브`로 굿즈코드 산출 = **0행**, 실제 `sync_itemdb` 기준 = **5,300/5,337행** 산출 가능.
- **영향:** 발주 표의 "굿즈코드" 항상 `—`, 매입추이(굿즈코드별)·제조유형×굿즈코드 매트릭스·매입비중(카테고리) 등 굿즈코드 기반 시각화가 전부 `기타` 한 덩어리로 붕괴.
- **수정:** `r['sync_itemdb']` 참조로 교체(정규식 `/([A-Z]{4})_/`는 `CTPS_PT4618...` 형식과 그대로 매칭됨).

### C3. 재고운영·EOQ·졸업 3개 페이지가 통째로 빈다
- **위치:** `renderInventoryOps`·`renderEOQ`·`renderGraduation` — 공통으로 `관리대상여부 === 'Y'` 필터.
- **원인:** 현재 `inventory_weekly.csv`에 **`관리대상여부`·`굿즈카테고리` 컬럼이 없음.**
- **데이터 검증:** 관리대상 Y 행 = **0 / 5,766**. → 재고회전율 0.00회, 품절률 0%, 졸업후보 0개, EOQ 발주필요 0, 카테고리 표 공란/미분류.
- **영향:** 상위레벨이 열어보는 재고 지표 3개 화면이 "데이터 있는데 0으로" 나옴 — 가장 오해 소지가 큰 유형.
- **선택 필요:** ① Airtable export에 두 컬럼 추가(권장, 근본 해결) vs ② 컬럼 부재 시 전 행을 관리대상으로 간주하도록 코드 폴백. → **A/B 질문으로 확인 예정.**

### C4. 헤더 매입금액 KPI가 실제보다 최대 20% 왜곡되어 보임
- **위치:** `fa()` L929~933 (천만 단위 반올림)
- **데이터 검증:** `fa(25,000,000)` → **"2천만"**(실제 2,500만, −20%), `fa(15,000,000)` → "2천만"(+33%).
- **영향:** 종합현황 매입금액·긴급 등 헤더 KPI가 눈대중으로 최대 수백만 원 어긋나 보임. 표(`faFull`)는 정확하므로 카드 값과 표 값이 불일치.
- **수정:** 천만 구간을 1자리 소수(예: "2.5천만")로.

## B. 치명적 — 공개 즉시 터질 배포 구조 (숫자와 별개)

### S1. 접속자는 매번 CSV 6개를 수동 업로드해야 화면이 채워짐
- **위치:** `window.addEventListener('load')` L4135 — `data_2025/cost_db/parts_master` JSON만 자동 fetch. **발주·이슈·공급망·고객인지·재고주간·매출월간 RAW는 자동 로드 코드가 없음.**
- **영향:** 사내 호스팅으로 배포해도, 링크를 연 상위레벨/구매팀원은 **빈 대시보드**를 보고 직접 6개 파일을 드래그해야 함. "빠르고 직관적으로 보고" 목적과 정면 충돌.
- **해결 방향:** 배포용으로 RAW를 `data/*.json`(또는 `CSV/*.csv`)에 넣고 init에서 자동 fetch. Airtable→가공→업로드 파이프라인과 직결되는 지점 — **별도 설계로 진행 제안.**

### S2. 자동인식은 헤더 정확 일치에 의존 → export 헤더 한 글자만 바뀌어도 조용히 실패
- **위치:** `detectCSVType` / `FILE_SIGNATURES` L1204~
- **영향:** Airtable 뷰 컬럼명이 바뀌면 해당 파일이 인식 실패로 빠지고, 경고 없이 그 섹션만 빈 채로 배포됨. 지금도 판매주문(`연월`/`소계`) 파일 미제공이라 매출/마진 페이지는 상시 "데이터 없음".

## C. 중간 — 정합성·견고성 (공개 전 정리 권장)

- **M1. 재고소유구분 상시 공란:** 품절리스트 컬럼은 `재고소유구분 (from sync_parts)`인데 코드는 `재고소유구분` 참조(L1648·L2576·L2601). → 품절표 "재고소유" 칸 항상 빈칸.
- **M2. YoY 뱃지 색상 의미 혼동:** `yoyBadge`는 증가=빨강/감소=초록 고정(L1402). 이슈엔 맞지만 **매입금액·발주건수 증가도 빨강**으로 나와 상위보고 시 "초록=감소"로 오독될 수 있음. 지표별 색 방향 분리 필요.
- **M3. 매입금액 24% 누락 캐비앗:** 발주 5,337행 중 세금계산서월 공란 **1,279행(24%)**은 매입 집계에서 제외됨(세금계산서 기준 설계). 의도된 동작이나, 카드에 "세금계산서 기준" 외 미발행분 규모를 병기하지 않으면 총액이 작아 보임.
- **M4. localStorage 캐시 무효화 없음:** `data_2025/cost_db/parts_master/kpi_targets/sp_confirmed/quote_history`에 버전키·만료가 없어, 오래된 캐시가 남고 fetch 실패 시 옛 데이터를 그대로 노출. 배포 갱신 시 강제 무효화 수단 필요.
- **M5. k-order KPI 정의 일관성 확인:** 종합 발주 KPI는 재고생산+프로젝트 합산인데 YoY 기준(2025 집계)의 정의가 동일한지 미확인. 전년 대비가 사과-오렌지 비교일 여지.

## D. 낮음 — 성능/코드 위생

- **L1. onclick 문자열 삽입:** 협력사명 등을 `onclick="...('${esc(x)}')"`에 넣는데 `esc`는 `& < >`만 처리. 따옴표/역슬래시 포함 이름에서 핸들러가 깨질 수 있음(일부는 `replace(/'/g)` 보강돼 있으나 불균일).
- **L2. 전량 재렌더:** 월 클릭·검색마다 `refreshAll()`이 모든 탭·차트를 재생성. 발주 5,000행+ 규모에서 체감 지연 가능. 필요한 섹션만 갱신하도록 분리 여지.
